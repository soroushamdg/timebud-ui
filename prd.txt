PART A ? PRODUCT REQUIREMENTS

1. What TimeBud Is
TimeBud is an AI task engine for students with ADHD. The user says how many hours they have to work. TimeBud tells them exactly what to do in that time ? across all their active projects, broken into small achievable steps.

Core value proposition: You tell TimeBud how long you have. It tells you exactly what to do.
TimeBud does not manage calendars, habits, sleep, or meetings. It owns exactly one thing: the inside of a work block the user has already carved out. It is not a scheduler. It is not a habit tracker. It is a session-level brain for people who know they have time to work but freeze when deciding what to do with it.

When a user adds a project they do not know how to approach ? for example, "build a mobile app for my class" ? TimeBud researches the best approach using Perplexity, generates a structured milestone plan using Claude, and breaks it into 10?25 minute tasks. The user never stares at a blank screen wondering where to start.


2. Problem Statement
Students with ADHD face two consistent failure modes when managing multiple projects:
? Task initiation paralysis ? they know a deadline exists but freeze when sitting down because they do not know what specific thing to do first.
? Project neglect ? they hyperfocus on one project and ignore others until multiple deadlines collide at once.

Existing tools fail this audience for two reasons. Tools like Motion and Reclaim manage the entire day ? meetings, habits, calendar ? which creates cognitive overload. Tools like Todoist and Notion require the user to already know what tasks need to be done, which is precisely the gap. Nobody builds the bridge between "I have a project due" and "here are your next 25 minutes of concrete work."

TimeBud closes this gap by owning only the inside of a work block, not the whole day.


3. Architecture
NextJS (Vercel)  ??  FastAPI (Railway)  ??  Supabase (PostgreSQL)  +  Claude API  +  Perplexity API
The system is two fully independent services communicating over HTTPS:
? FastAPI (backend): all business logic, AI integrations, the session planning algorithm, and the Supabase-hosted PostgreSQL database. Deployed on Railway. Exposes a clean REST API. Auto-generates a Swagger UI at /docs for manual testing during development.
? NextJS (frontend): all UI, routing, and auth via Clerk. Deployed on Vercel. Contains zero business logic. Every user action calls a FastAPI endpoint. The TypeScript API client is auto-generated from FastAPI's /openapi.json schema using openapi-ts.

This separation means a Flutter mobile app in Phase 2 calls the same FastAPI endpoints with no backend changes. Frontend and backend are developed, tested, and deployed independently.

3.1 Authentication Flow (Clerk)
Authentication is handled entirely by Clerk. The frontend uses Clerk's drop-in React components. FastAPI verifies Clerk JWTs by fetching Clerk's public JWKS endpoint at startup ? no secrets are shared between services.

StepWhereWhat Happens1NextJS + ClerkUser sees Clerk's drop-in <SignIn /> component wrapped in <ClerkProvider>. Clerk handles the Google OAuth flow, UI, and session management.2Clerk SDKOn successful sign-in, Clerk issues a short-lived JWT (session token) containing the user's Clerk user_id, email, and metadata.3NextJS ? FastAPIEvery API request from NextJS attaches the Clerk JWT as a Bearer token in the Authorization header. Obtained via Clerk's useAuth() hook or getToken() server-side.4FastAPI middlewareOn startup, FastAPI fetches Clerk's JWKS public keys. On every request, the auth middleware decodes and verifies the JWT signature using PyJWT + the cached JWKS. Fully stateless ? no session table needed.5FastAPI DBOn the first verified request for a user, FastAPI upserts a user row in Supabase keyed on the Clerk user_id.Clerk's free tier covers unlimited Monthly Active Users (MAUs) for personal/hobby projects, making it $0 at MVP scale.

3.2 TypeScript Client Generation (openapi-ts)
Instead of manually writing and maintaining lib/api.ts, the TypeScript API client for FastAPI is auto-generated using openapi-ts. FastAPI natively exposes its full schema at /openapi.json. openapi-ts reads this schema and outputs a fully-typed client with one command:
npx openapi-ts --input https://your-api.railway.app/openapi.json --output src/client

This means every time a FastAPI endpoint changes ? a new field, a new route, a renamed parameter ? the frontend client is regenerated in seconds. Type errors surface immediately at compile time rather than at runtime. No manual API wrapper code to write or maintain.
Re-run the generation command whenever the FastAPI schema changes during development.

5.1 Project Setup ? GTD-Guided Form
The new project flow at /projects/new shows one question per screen. ADHD users are overwhelmed by long forms. Each screen has a single input, a progress indicator (dots at the top), and a Next button.

5.2 Work Session ? The Core Feature
The session screen is the heart of TimeBud. It is stripped of everything except the current task.

Session Setup
User navigates to /session/new. One large input: "How long can you work right now?" Optional toggle to exclude specific projects. Tapping Start calls POST /sessions on the FastAPI backend.

Planning Algorithm
The backend computes a time allocation across all active projects using a weighted priority formula:
weight = priority_score  x  urgency_multiplier  x  remaining_work_factor
priority_score:       High = 3  |  Medium = 2  |  Low = 1
urgency_multiplier:   exponential -- doubles as days_to_deadline halves
remaining_work:       estimated_minutes_remaining / total_estimated_minutes

Output: a flat ordered list of micro-tasks (10?25 minutes each) filling the user's time budget. Tasks from multiple projects are interleaved by weight ? the user works through one list without thinking about project switching.

Session UI
? Full-screen task card: task title, estimated duration, subtle project label in the corner
? Optional countdown timer: user can toggle on or off
? Three buttons only: Done  /  Skip  /  How?
? Done: marks task complete, brief animation, next task slides in
? Skip: queues task for next session, advances immediately
? How?: bottom drawer with Claude's step-by-step explanation. Does not navigate away from the session screen.
? Thin progress bar at top: tasks completed / total tasks in session
? Remaining time shown small in the corner ? present but not dominant

Re-planning
TriggerTimeBud ResponseAll tasks completed earlyCalls POST /sessions/{id}/replan. Offers bonus tasks or ends session with a celebration.Session time runs outGentle prompt: "Time's up ? keep going or wrap up?" Calls POST /sessions/{id}/end if wrapped.Task skippedPATCH /tasks/{id}/skip queues task for next session and updates its priority weight.Deadline is tomorrowRed urgency tint on project card on /home. Browser notification if permission granted.
5.3 UI Design Rules
? One primary action per screen: no equal-weight option menus. The dominant next action is always obvious.
? Progressive disclosure: project cards show only name and deadline. Tasks only appear during a session.
? Generous whitespace: no dense lists. Items breathe. Large tap targets (minimum 44px height).
? No dashboards in v1: no charts, streaks, heatmaps, or statistics.
? Session mode is fullscreen: when a session is running, the nav bar and all other UI disappears. One task. Three buttons.
? Colour is functional: colour signals urgency (red = deadline close) or completion (green). Never used decoratively.

7. Data Model (Supabase / PostgreSQL)
The schema is hosted in Supabase PostgreSQL from Day 1. No migration from SQLite is ever needed. SQLAlchemy async ORM connects FastAPI to Supabase using the connection string from the Supabase project dashboard.
Connection string format: postgresql+asyncpg://postgres:[password]@db.[project-ref].supabase.co:5432/postgres

users
id               SERIAL      PRIMARY KEY
clerk_user_id    TEXT        UNIQUE NOT NULL   -- Clerk identity key (replaces google_sub)
email            TEXT        NOT NULL
name             TEXT
created_at       TIMESTAMPTZ DEFAULT NOW()

projects
id               SERIAL      PRIMARY KEY
user_id          INTEGER     REFERENCES users(id) ON DELETE CASCADE
name             TEXT        NOT NULL
description      TEXT
deadline         TIMESTAMPTZ
priority         TEXT        CHECK(priority IN ('high','medium','low'))  DEFAULT 'medium'
knows_steps      BOOLEAN     DEFAULT FALSE
status           TEXT        DEFAULT 'active'   -- active | archived
created_at       TIMESTAMPTZ DEFAULT NOW()

milestones
id               SERIAL      PRIMARY KEY
project_id       INTEGER     REFERENCES projects(id) ON DELETE CASCADE
title            TEXT        NOT NULL
description      TEXT
display_order    INTEGER     NOT NULL
status           TEXT        DEFAULT 'pending'  -- pending | done
estimated_hours  NUMERIC(6,2)

tasks
id               SERIAL      PRIMARY KEY
milestone_id     INTEGER     REFERENCES milestones(id) ON DELETE CASCADE
title            TEXT        NOT NULL
estimated_minutes INTEGER
status           TEXT        DEFAULT 'pending'  -- pending | done | skipped
completed_at     TIMESTAMPTZ

sessions
id               SERIAL      PRIMARY KEY
user_id          INTEGER     REFERENCES users(id) ON DELETE CASCADE
date             DATE        NOT NULL
